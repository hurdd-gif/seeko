## Design Context

### Users

SEEKO Studio is used by admins and investors. Admins need to operate the studio, triage work, review progress, and safely make changes. Investors need a clear, trustworthy read on studio health without being exposed to internal implementation noise. The in-dashboard agent is a personal studio companion: it should help users understand, prepare, summarize, and act without feeling like a generic chatbot.

### Brand Personality

Quiet, precise, companionable. The product should feel like a calm studio operating system: restrained, confident, and useful under pressure. It should not feel like a sci-fi AI console, a marketing assistant, or a playful toy.

### Aesthetic Direction

Use SEEKO's existing light-dashboard direction and Fluid Functionalism-inspired component behavior: refined surfaces, satisfying hover states, visible state transitions, and motion that explains what changed. The assistant should live as a minimal bottom-right companion, not a full-height panel. Its default state should be a compact dock that expands into a small contextual tray only when the user asks or when an approval needs attention. The latest visual direction is frosted, glassy, and modern-forward: a restrained dark translucent surface is acceptable for the companion because it separates the agent from the light dashboard without becoming a sci-fi console. The assistant should use existing Inter-led typography, `#0d7aff` action accents, warm approval accents, and risk/error colors already present in the dashboard.

### Design Principles

1. Make agent work visible: show thinking steps, tool calls, approvals, and completed actions as understandable interface states.
2. Keep trust above speed: risky actions require explicit approval before writes, while low-risk reads and drafts can run immediately.
3. Separate audiences: admins can act; investors can ask, summarize, and inspect but should not receive write controls.
4. Preserve dashboard calm: avoid novelty AI styling, heavy glow, gradient spectacle, large chat drawers, and excessive cards.
5. Design for 24/7 triggers: user chat, webhook events, cron runs, and background jobs should all resolve into the same run history and audit model.
